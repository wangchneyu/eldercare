## 一、命名、模型与分层

1. 类使用 `UpperCamelCase`；方法、变量使用 `lowerCamelCase`；常量使用全大写下划线风格。
2. 包名仅使用小写英文单词和点号，不使用横线、下划线、拼音或数字开头。正式组织域名未确定前，基础包统一使用占位格式 `com.<organization>.<project>.<domain>`，例如 `com.<organization>.<project>.care`；公共包使用 `com.<organization>.<project>.common.<module>`。组织域名确定后，应一次性替换全部占位段。
3. 仅使用 `DTO`、`VO`、`JWT`、`MQ`、`SOS`、`ID` 等通用缩写；自定义缩写须在首次出现处说明。

| 类型 | 后缀 | 使用边界 |
| --- | --- | --- |
| 持久化实体 | `Entity` | 仅 Mapper/Repository 和 Service 内部使用，不得作为 REST 或 Feign 入出参 |
| REST 请求对象 | `Request` 或 `DTO` | Controller 入参，必须声明校验注解 |
| REST 响应对象 | `VO` | 对客户端返回；不得暴露密码、Token、删除标志及内部实现字段 |
| 查询对象 | `Query` | 仅承载分页、筛选条件，不得承载写操作字段 |
| 服务契约对象 | `RemoteDTO` | 仅用于 `common-feign` 的同步查询契约 |
| 消息对象 | `Event` | 仅用于 MQ 消息体，必须版本化 |

对象转换集中在 assembler/converter；不得在 Controller、Mapper 或接口边界散落复制逻辑。

1. Controller 只做鉴权、参数校验、协议转换和调用应用服务；不得直接访问 Mapper，不得实现状态机、重试或跨服务编排。
2. Service 接口使用 `IXxxService`，实现类使用 `XxxServiceImpl`。本地事务在 Service 层定义；事务中禁止远程调用、文件上传和 AI 推理。
3. Mapper 方法以 `select`、`insert`、`update`、`delete` 开头；逻辑删除必须在方法名或注释中说明。

## 二、跨服务调用与消息

1. 跨服务写操作、状态变更和业务编排默认通过 RocketMQ 事件实现最终一致性，禁止通过 Feign 串行调用构造跨服务事务。确有强一致性需求时，须经架构评审后采用 Seata TCC/XA 或 Saga 补偿方案；不得用于 P0 实时链路及包含外部副作用的操作。
2. Feign 仅允许用户请求链路中的只读、低延迟查询，且调用方能明确反馈“暂不可用”或展示带时间戳的缓存数据。P0 处置、扣费、库存、核销和权限判定禁止以 fallback 的默认值继续执行。
3. AI 推理是受超时与熔断保护的同步例外；P0 意图识别必须有同进程规则引擎兜底，不能静默降级为成功。
4. Feign 契约统一放在 `common-feign`，只包含接口、`RemoteDTO`、配置与明确 fallback；服务端 Controller 不依赖该模块。Feign 不返回 REST 的 `R<T>`。
5. Feign 超时、重试、日志级别只在 Nacos 维护。默认连接超时 2 秒、读取超时 5 秒；例外必须登记调用方、超时预算、重试条件和降级行为。

所有 MQ 事件必须携带以下信封字段：

| 字段 | 说明 |
| --- | --- |
| `eventId` | 全局唯一事件 ID，也是消费幂等键 |
| `eventType` / `schemaVersion` | 事件名称和版本；新增字段必须向后兼容 |
| `occurredAt` | UTC ISO-8601 产生时间 |
| `traceId` | 贯穿 HTTP、Feign、MQ 与日志 MDC |
| `producer` | 服务名及版本 |
| `payload` | 业务数据，禁止密码、Token、完整证件号码等敏感字段 |

1. 关键事件须与本地数据库变更原子衔接：使用 RocketMQ 事务消息或 transactional outbox，二选一并在服务方案中说明。`syncSend` 成功不代表端到端业务成功。
2. P0 消息必须定义端到端超时预算、发送失败告警、消费重试、死信后的人工升级和补偿任务；重试不得突破 SOS 链路时限。
3. 幂等以数据库唯一约束、消费记录表或状态机条件更新为准。Redis 锁仅用于并发优化，不能作为唯一正确性保障；去重保留期须覆盖补偿窗口。
4. 消费失败分为可重试临时故障、不可重试数据/业务错误和人工处理。死信必须保留原始事件、失败原因和 `traceId`。

## 三、异常、日志与可观测性

1. 可预期业务规则失败使用 `BizException`，且只能引用已登记的 `IErrorCode`；不得以 `RuntimeException` 表达业务失败。
2. 参数校验、认证授权、反序列化、数据库、Feign、MQ 和第三方 SDK 异常由基础设施层或全局处理器分类处理；不得一律包装成业务异常。
3. 客户端消息必须是错误码绑定的稳定、脱敏文案；禁止拼接用户输入、个人信息、SQL、内部地址和第三方原始异常。排障上下文写结构化日志并脱敏。
4. 每个入站请求生成或透传 `traceId`，写入 MDC 和响应体；Feign Header、MQ Header 和异步线程池必须透传并在任务结束后清理 MDC。
5. 业务异常以 WARN 记录错误码、路径和脱敏上下文；未知异常以 ERROR 记录完整堆栈。禁止 `System.out`，统一使用 SLF4J。

## 四、数据库与安全

1. 业务主键使用雪花 ID：数据库为 `bigint`、Java 为 `Long`；REST 和 MQ JSON 中的 ID 必须序列化为字符串，避免 JavaScript 精度丢失。
2. 跨服务关联字段统一为：`elder_id bigint`、`worker_id bigint`、`device_id varchar(64)` 或经模型评审的数值类型；禁止跨库、跨服务 SQL JOIN。
3. 时序表的时间列必须进入主键或唯一索引，并作为索引前导列；实际分区策略以 TimescaleDB 建表规则为准。
4. JSONB 仅存低频变化、非核心检索属性。索引按查询模式选择：包含/存在查询可用 GIN，单键等值或排序优先评估表达式 B-tree；每个索引必须对应实际查询。
5. HMAC 行签名必须使用确定性字段序列化，并包含 `keyId`、主键、关键状态、关联主体和创建时间。密钥轮换必须能校验历史记录；验签失败不得静默读取或覆盖，应审计并触发高优先级告警。
6. 密钥、Token、证书和第三方凭据只能从受控加密配置注入；禁止硬编码、写日志或发送到 MQ。

## 五、质量门禁与禁止项

1. 公开 API、Feign 契约、MQ 事件和错误码变更必须有契约测试；状态机、幂等、重试、事务消息和异常映射必须有自动化测试。
2. 新增错误码须先登记；禁止复用或改变既有错误码语义。删除功能时标记废弃，不得重新分配。
3. 禁止无超时 HTTP/Feign 调用、无界线程池、未同步共享可变集合、吞掉异常后返回成功、在事务中调用远程服务。
4. 注释解释“为什么”和边界条件。TODO 必须关联任务编号与截止时间，禁止遗留无主 TODO。
