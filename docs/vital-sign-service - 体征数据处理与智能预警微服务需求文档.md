## 1. 文档基本信息

- **项目/功能名称**：高并发物联网（IoT）生理体征接入、流式预警计算与多维历史趋势查询系统
- **文档版本**：v1.0.0
- **创建/修改时间**：2026-07-20
- **产品经理(PM)**：系统架构师组
- **主要开发(Owner)**：周光照（体征服务开发负责人）
- **下游/关联方干系人**：-
  - `iot-service`（硬件网关，上游数据流生产者）
  - `care-service`（业务系统/规则配置，上游规则生产者）
  - `alert-service`（告警触达，下游事件流消费者）
  - 前端/移动端研发团队（家属 App & 护士站大屏看板）

## 2. 业务背景与目标

- **为什么要做这个功能**： 康养院区统一部署了 500+ 台智能床垫、智能手表等硬件设备，每秒高频上报长者的心率、呼吸、体动等生理指标。每日产生高达 $40,000,000$ 条（年均近 150 亿条）时序状态数据。

  传统的单体 CRUD 架构无法承受如此庞大的高并发吸纳，且直接利用关系型数据库进行最新状态读取及跨月历史趋势拉取会导致磁盘 I/O 瞬间打爆。我们需要建立一套兼顾**高吞吐量写入**、**亚毫秒级实时快照读取**、**防噪声抖动（防毛刺）流式实时计算**与**动态降采样多模趋势查询**的工业级时序数据流管线。

- **业务价值与期望达成的事后指标**：

  1. **极速吸纳**：500 次/秒并发写入 TimescaleDB，底层单条写入耗时控制在 $1\text{ms}$ 以内。
  2. **实时大屏响应**：护士站大屏/App 首页查询“当前最新一秒状态”实现读写隔离，Redis 快照查询延迟低于 $1\text{ms}$。
  3. **零延迟告警通道**：实现微服务高内聚，下游 `alert-service` 接收到告警事件后，**免反查关系型数据库**，组装并发送短信延时由秒级缩短至 $10\text{ms}$ 内。
  4. **防误报平滑性**：引入内存滑动窗口防噪算法，排除因长者翻身、传感器静电等单点抖动造成的“狼来了”虚假告警，告警准确率提升至 $99\%$。
  5. **前端丝滑渲染**：大跨度（如 2 年内）历史折线图查询加载时间由几分钟降至 $100\text{ms}$ 以内，渲染点数恒定在 $1,000$ 个以内的黄金流畅区。

## 3. 服务边界与职责定义

```
                  ┌──────────────┐
                  │ care-service │
                  └──────┬───────┘
                         │ (1) 广播阈值规则变更 (MQ Rule Update)
                         ▼
┌─────────────┐   ┌──────────────────────────────┐ (4) 触发超阈值胖报文  ┌───────────────┐
│ iot-service ├──►│     vital-sign-service       ├────────────────────►│ alert-service │
└─────────────┘   └──────────────┬───────────────┘ (MQ Alert Event)     └───────────────┘
 (2) 极简时序流                   │             │
   (MQ Raw Data)                  ▼             ▼
                          ┌──────────────┐┌────────────┐
                          │ TimescaleDB  ││   Redis    │
                          └──────────────┘└────────────┘
                            (历史时序归档)  (实时状态快照)
```

- **本服务 (vital-sign-service) 负责的逻辑**：
  - **时序数据流摄入**：监听 RocketMQ 中 `iot-service` 发送的秒级极简物理时序流，将其高速异步写入 TimescaleDB。
  - **规则内存热加载**：无需每次查询数据库，通过本地 JVM 内存（`ConcurrentHashMap`）实时加载并热更新 `care-service` 广播的动态生命体征规则。
  - **流式预警计算**：内置高性能滑动窗口算法与状态机，对每一秒摄入的数值进行内存比对与防毛刺去噪，确认真实的告警事件。
  - **事件胖报文发射**：一旦判定异常，在内存中瞬间“撞库”补全长者姓名、床位号等业务元数据，向 `alert-service` 发射携带唯一 `event_id` 的高自包含“胖报文”。
  - **实时快照同步**：每次收到数据时，将最新的指标极速覆盖写入 Redis，并配备“设备离线 TTL 哨兵”。
  - **双模式 API 提供**：暴露历史体征查询接口，针对大跨度时间范围在后端执行智能路由，实现原始表与连续聚合视图的切换。
- **其他相关服务负责的逻辑 (外部依赖)**：
  - **`iot-service`**：负责硬件的长连接接入、协议解析、屏蔽底层物理异构性，向 MQ 投递**仅携带 `device_id` 与时间戳、绝对物理客观**的极致骨感报文。
  - **`care-service`**：作为业务主库的守护者，负责“长者-房间-设备”静态绑定关系的维护、临床阈值规则的界面化配置，并在变更时向 MQ 发送广播。
  - **`alert-service`**：纯粹的告警接收与外呼通道。负责消费体征告警事件流，必须在自身数据库/缓存中利用 `event_id` 进行强幂等性（去重）拦截，防止因网络重试引发家属恐慌。
- **不属于本次范围 (Out of Scope)**：
  - 长者基本信息与设备关系的 CRUD 后台配置界面（属于 `care-service` 范围）。
  - 硬件通信协议调试、丢包重发逻辑（属于 `iot-service` 范围）。
  - 实际短信网关、电话网关的对接和路由调度（属于 `alert-service` 范围）。

### 4.2 详细业务规则

#### 规则 1：事实与维度解耦原则（运行时内存绑定）

- **物理现实**：时序原始表 `smart_mattress_raw_data` 每天写入 4000 万条记录。
- **架构规避**：严禁在此表中冗余 `elder_id`（长者 ID）、`elder_name` 或 `device_type`。
  - *原因*：每天会产生约 1.2 GB 的纯冗余垃圾数据，导致 Buffer Pool 内存污染。且一旦发生“长者换床位/设备损坏更换”，将引发百亿级数据的灾难性 `UPDATE`。
- **执行方式**：系统运行时，通过监听 `device_registry` 的变更，在 `vital-sign-service` 的 JVM 内存中维护一个高性能的 `device_id -> elder_id` 的常驻 Map，进行 $O(1)$ 级别极速撞库。

#### 规则 2：流式防噪过滤规则（防传感器毛刺）

- **痛点**：长者夜间翻身或静电可能导致某一秒心率突升至 160，若单点超标立即告警将造成“狼来了”的疲劳效应。
- **判定算法**：流计算引擎采用“滑动窗口（Sliding Window）”过滤策略。
  - *逻辑*：以 15 秒（5 次采集点）为计算窗口。当窗口内**异常点占比超** $80\%$ **且持续超过 15 秒**，才判定为“真实异常”并向外发射告警事件；单点抖动直接在内存中静默吸收。

#### 规则 3：Redis 实时快照过期时间（离线哨兵机制）

- **痛点**：若床垫由于停电或网络故障断开连接，Redis 中的数值如果不清理，大屏和 App 会永远显示离线前的最后状态（例如“心率 75”），产生巨大安全隐患。
- **防御规则**：每次写入 `vital_sign:snapshot:elder:{elder_id}` 时，强制对该 Key 附加 `EXPIRE` 10 分钟。一旦设备断网超时，该 Key 在 Redis 中自动蒸发。前端查询为空时自动将其渲染为 `⚠️设备离线`。

### 5.2 异步消息契约（MQ）

#### 5.2.1 消费时序流（Subscribe）

- **数据生产者**：`iot-service` (硬件数据网关)

- **Topic / Tag**：`iot-vital-raw-topic` / `mattress`

- **消息体格式**：JSON

- **设计理念**：**极致骨感**。为节省每秒 500 次写入的网络带宽，该报文抛弃一切业务修饰，KEY 全部精简，仅包含物理世界的绝对事实。

- **Payload 契约 Schema**：

  ```
  {
    "device_id": "BED-001",
    "ts": 1781830200000,
    "hr": 75,
    "rr": 18,
    "bm": 2,
    "status": 1
  }
  ```

  *(说明：ts: 毫秒级时间戳; hr: 原始心率; rr: 原始呼吸率; bm: 体动强度 0-10; status: 在床状态 1=在床, 2=离床, 3=离床超时)*

## 6. 数据架构与存储变动

### 6.1 时序数据超表 (TimescaleDB) DDL 定义

抛弃无意义且高并发下引发严重锁争抢的 `BIGSERIAL` 自增主键。直接使用物理世界上天然而唯一的 `(device_id, data_time)` 作为联合主键。

```
-- 创建极致骨感的体征原始事实表
CREATE TABLE smart_mattress_raw_data (
    device_id VARCHAR(32) NOT NULL,
    data_time TIMESTAMPTZ NOT NULL,
    heart_rate SMALLINT DEFAULT NULL,         -- 数据降级，省50%存储，完全满足心率表示范围
    respiratory_rate SMALLINT DEFAULT NULL,   -- SMALLINT (2字节) 代替 INTEGER (4字节)
    body_movement SMALLINT DEFAULT NULL,
    bed_status SMALLINT DEFAULT NULL,         -- 数字枚举替代VARCHAR(16)，提升10倍列式压缩率
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (device_id, data_time)
);

-- 启用 TimescaleDB 核心超表特性，【仅按时间一维分区】（一维时间分区大Chunk是单节点抗写最强解）
SELECT create_hypertable('smart_mattress_raw_data', 'data_time', chunk_time_interval => INTERVAL '1 day');

-- 配置冷热分离：30天冷数据自动列式压缩，开启 segmentby 数据聚簇
ALTER TABLE smart_mattress_raw_data SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'device_id',  -- 按照设备ID在压缩内归拢，大跨度查单个设备快百倍
    timescaledb.compress_orderby = 'data_time'
);

-- 添加自动压缩定时任务 (30天后自动压缩)
SELECT add_compression_policy('smart_mattress_raw_data', INTERVAL '30 days');

-- 添加数据保留策略 (原始明细数据保留 90 天，到期自动 DROP CHUNK，零碎片且不占 I/O)
SELECT add_retention_policy('smart_mattress_raw_data', INTERVAL '90 days');
```

## 7. 非功能性需求 (NFR)

### 7.1 性能与容量评估

- **吞吐量支撑**：峰值吞吐量预计 $1,000+\text{ QPS}$，平均 $462\text{ QPS}$。通过 TimescaleDB 的追加写设计，主键索引更新吞吐无瓶颈。
- **I/O 与缓存污染防控**：因为剔除了 `elder_id` 这类大量重复元字符串，时序数据列式压缩率提升至近 $90\%$。单节点物理磁盘空间由每年原预计消耗的约 $800\text{ GB}$ 降至仅 $90\text{ GB}$，真正做到高能降本。
- **连续聚合 (Continuous Aggregates)**： 对于大跨度查询视图的拉取，TimescaleDB 连续聚合定时刷新的数据直接存储在聚合视图表中（例如 1 天的聚合仅产生 1 条数据）。数据库通过合并聚合结果和近期的内存热数据进行“实时拼接”，实现毫秒级快速拉取。

### 7.2 稳定性与防御策略（服务韧性）

- **连接池读写分离与写入优先原则**： 由于写入操作是绝对的“生命线”，本服务在底层配置两个完全隔离的关系型连接池：
  - **写入连接池 (Write Pool)**：最高优先级。连接数上限扩容，写入链路绝不可因为查询请求堆积而发生阻塞。
  - **查询连接池 (Read Pool)**：面向 App 与护士站。若查询阻塞，绝不影响写入，做到完全物理隔离。
- **超期死信队列 (DLQ) 监控即时熔断策略**： 对于告警事件流，若网络重试 16 次均告失败而掉入 RocketMQ 的死信队列，**严禁使用每小时自动脚本去批量重录**。因为医疗预警具有极高的时间敏感度，1 小时前的旧警告已彻底丧失急救价值。
  - *落地方案*：配置专属的死信队列阈值报警。一旦告警消息触及死信队列，直接触发 P0 级最高等级运维电话外呼告警，人工紧急介入，对宕机节点或 Bug 进行即时熔断复位。

### 7.3 安全与合规

- **接口防爆盾限流**： `GET /history` 接口，单机最大并发查询上限限制为 `100`。超出直接返回 `429 Too Many Requests`，保护主库 CPU 不被因医生大跨度高频框选查询而被耗尽。
- **鉴权安全**： 所有 API 请求必须通过网关层剥离的 JWT Token 进行严格鉴权。
- **数据脱敏**： 微服务日志库输出中，严禁打印涉及长者手机号、银行卡或完整住址的字段信息。
